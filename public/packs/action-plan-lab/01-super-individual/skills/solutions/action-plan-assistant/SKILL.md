name: action-plan-assistant
description: 单 Agent 帮用户把一个目标、会议结论或讨论结果直接压成"可执行、可发送的行动计划表"。
---

# 什么时候触发
当用户已经有一段会议纪要、项目目标、头脑风暴记录或零散想法，希望不要停在原始记录，而是先拿到一版有负责人、截止时间、优先级和依赖关系的可执行行动计划时触发。

# 目标
把原始输入压成：
- 目标摘要
- 动作项（具体要做什么）
- 负责人（谁来做）
- 截止时间（什么时候交）
- 优先级（P0/P1/P2）
- 依赖关系（哪些动作之间有先后）
- 风险与阻塞提醒
- 发送版行动计划正文
- 下一句续跑指令

# 输入
- 项目/会议/讨论主题
- 参与人 / 角色
- 原始记录（会议纪要 / 讨论记录 / 项目目标 / 零散想法）
- 当前约束（哪些已定、哪些不能误判、输出格式偏好）

# 输出
至少输出下面 9 类内容：
1. 目标摘要
2. 动作项清单
3. 负责人分配
4. 截止时间
5. 优先级排序
6. 依赖关系图
7. 风险与阻塞提醒
8. 发送版行动计划正文
9. 下一句续跑指令

# 最小步骤
1. 先还原目标和背景
2. 区分已决策、待确认和未决事项
3. 拉出动作项并尽量补 owner / deadline / priority
4. 排出依赖关系和执行顺序
5. 补风险和阻塞提醒
6. 压成发送版行动计划正文
7. 最后补下一句续跑指令

# 强制规则
- 不要把讨论误写成已决策
- 不要把待确认事项误写成已定动作
- 如果原始记录里出现 owner / deadline，要优先显式抽出
- 每个动作项至少包含：动作 + 负责人 + 截止时间 + 优先级
- 如果信息不完整，要明确写"待确认"，不要擅自补定案
- 输出必须是能直接发送的，不只是建议

# 常见坑
- 只给总结，不给动作项
- 已决策和待确认混在一起
- 漏 owner、漏截止时间、漏优先级
- 不排依赖关系，导致执行顺序错误
- 语气太像分析报告，不像可发群的任务表

# 验证方式
用 sample-input 跑一次，检查输出是否同时包含：
- 目标摘要
- 动作项清单
- 负责人
- 截止时间
- 优先级
- 依赖关系
- 风险提醒
- 发送版正文
- 下一句续跑指令
