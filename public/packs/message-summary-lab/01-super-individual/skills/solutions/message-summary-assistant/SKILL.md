# 什么时候触发
当用户已经有一封长邮件、一堆飞书/企微/钉钉群消息或跨部门讨论记录，希望不要逐条翻看，而是先拿到一版结构化、可转发、可同步的摘要时触发。

# 目标
把原始消息压成：
- 核心结论（到底在说什么）
- 关键信息点（3-5 条最重要的信息）
- 待办事项（谁承诺了什么，谁要做什么）
- 关键时间点（截止日期、会议时间、里程碑）
- 风险与变更提醒
- 发送版摘要正文（一段可直接转发）
- 下一句续跑指令

# 输入
- 消息来源（邮件 / 飞书 / 企微 / 钉钉 / 混合）
- 原始内容（全文 / 截图转写 / 复制粘贴）
- 用户角色（方便标注哪些跟用户相关）
- 当前约束（输出格式偏好、需要重点关注的人或事）

# 输出
至少输出下面 7 类内容：
1. 一句话核心结论
2. 关键信息点（3-5 条）
3. 待办清单（标注负责人）
4. 关键时间点
5. 风险与变更提醒
6. 发送版摘要正文
7. 下一句续跑指令

# 最小步骤
1. 先区分闲聊和关键信息
2. 提炼一句话核心结论
3. 拉出关键信息点
4. 提取待办事项并标注负责人
5. 标出关键时间点
6. 补风险和变更提醒
7. 压成发送版摘要正文
8. 最后补下一句续跑指令

# 强制规则
- 不要把闲聊误写成关键信息
- 不要把待确认事项误写成已定结论
- 如果消息里出现 deadline / 会议时间，要优先显式抽出
- 待办事项必须标注负责人，没有负责人的写"待认领"
- 输出必须是能直接转发的，不只是信息罗列

# 常见坑
- 只给原文节选，不给结构化摘要
- 关键信息和闲聊混在一起
- 漏 deadline、漏会议时间
- 不标注负责人，待办变成"大家注意"
- 语气太像信息整理，不像可转发的摘要

# 验证方式
用 sample-input 跑一次，检查输出是否同时包含：
- 核心结论
- 关键信息点
- 待办清单
- 关键时间点
- 风险提醒
- 发送版摘要
- 下一句续跑指令
